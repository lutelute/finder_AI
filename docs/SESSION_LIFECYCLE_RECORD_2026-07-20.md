# Terminalセッション管理の検証・設計記録（2026-07-20）

## 背景

FinderAIのウインドウやTerminalタブを消す操作と、Shell／Codex／Claudeの実行を終了する操作が近すぎた。タブを閉じると作業も失い、作業を残そうとするとタブを整理できないため、ユーザーが「表示」と「実体」のどちらを操作しているか判断しにくい状態だった。

実機確認では、終了操作後に一覧から消えたinteractive shellがFinderAI配下に残り、後から`<defunct>`になる事象も再現した。SwiftTerm 1.14の終了処理はSIGTERM後にprocess monitorを先に解除するため、SIGTERMを無視するshellが後から終了すると`waitpid`されない。FinderAI所有のPTY process groupへHUPを送り、応答しない場合はKILLし、専用queueで必ず回収する処理を追加した。

ただし、今回の中心課題はFinderAIプロセスを常駐させることではない。FinderAIのウインドウがすべて閉じても、セッション実体を見失わず、明示的に再表示・記録・終了できる管理面を作ることとした。

## ライフサイクルの分離

セッションには次の独立した操作を持たせる。

1. **タブを隠す**: UIからだけ除外する。マネージャの所有、PTY、Terminalバッファは維持し、プロセスへ終了信号を送らない。
2. **タブに表示**: 隠した同一セッションを再利用する。同じフォルダ・同じ種類のPTYを重複作成しない。
3. **記録を保存**: ユーザーが保存先を選んだ場合だけ、現在のTerminal表示バッファをテキストへ保存する。
4. **セッションを終了**: 確認後にPTYを終了してマネージャから除去する。永続セッションではFinderAI名義の対応tmuxセッションも終了する。

## 管理面

- 各タブの右クリックから「隠す」「記録を保存」「すべてを管理」「終了」を選べる。
- Terminalヘッダの管理ボタン、または`⌘⌥T`で全体管理画面を開ける。
- 管理画面はアプリ内セッションを「表示中」「バックグラウンド」「終了」に分け、tmux残存セッションも同じ一覧へ出す。
- 管理画面の「タブに表示」は対象フォルダへ移動してTerminalを展開する。tmux残存なら既存セッションへ再接続する。
- 永続化設定は新規セッションの作成方針であり、設定をオフにしても既存のFinderAI名義tmuxセッションを一覧・再接続・終了の対象から外さない。
- `finderai-`で始まらないユーザー所有のtmuxセッションには触れない。

## v1.4追記 — プロセスと独立した台帳

ライブPTYを所有する`TerminalSessionManager`とは別に、安定IDとフォルダ・種類・backend・時刻・表示／終了状態だけを持つ`TerminalSessionRecord`を導入した。実アプリはApplication SupportのJSONへ原子的に保存し、再起動後の管理画面にも履歴を表示する。Terminal内容は台帳に含めず、明示的な記録保存と区別する。decode不能な台帳は削除せず隔離し、FinderAIの起動を止めない。ライブPTYまたは実在tmuxと一致する履歴は一覧で重複排除する。
